# app/

The React + Vite + TypeScript PWA ŌĆö the SQL practice tool itself. A mature client-side tool: DuckDB-Wasm engine, **~70 questions across 6 datasets** (12 Snowflake), 15+ packs, output-equivalence grading with result-diff, worksheet editor, learning tracks + concept/dialect filters, progress/review/streak, adaptive next-question, PWA, a dev feedback loop, and a SnowflakeŌåÆDuckDB transpile layer. Phases 0ŌĆō4 largely done. See repo `ROADMAP.md` (live status) and `decisions/`.

## Setup
Uses **pnpm** (pinned via `packageManager`). Enable it with Node's corepack: `corepack enable && pnpm install`. **`npm install` fails** ŌĆö `apache-arrow` (transitive via duckdb-wasm) ships a broken `bin` entry that crashes npm but only warns under pnpm.

## PWA / offline
`vite-plugin-pwa` (config in `vite.config.ts`) generates a service worker + `manifest.webmanifest` on `pnpm build`. App shell is precached; the large DuckDB `.wasm` is **runtime-cached** (CacheFirst, `duckdb-wasm` cache) rather than precached, so install stays light and the app works offline after one real load. Icon is `public/icon.svg` (PNG 192/512 are a TODO for install prompts). For a subpath deploy, set Vite `base`. SW only active in `build`/`preview`, not `dev`.

## Dev feedback loop (closed-loop, dev-only)

- **Write side:** `vite/dev-feedback.ts` ŌĆö a Vite `apply:'serve'` plugin exposing `POST /__dev/feedback` and `/__dev/events`, writing to `<repo>/.dev-data/feedback.sqlite` (override via `FEEDBACK_DB`). Uses better-sqlite3, lazy-imported + guarded (disables gracefully if the binding is missing).
- **App side (dev only, `import.meta.env.DEV`):** `src/dev/telemetry.ts` (`logEvent`/`sendFeedback` + global error & click capture) and `src/dev/FeedbackWidget.tsx` (corner ­¤Æ¼ button ŌåÆ one submission bundles an optional ­¤æŹ/­¤æÄ with a note, sent only on Submit, auto-tagged with the current question). Logged events: session_start, nav, question_view, run, submit, reveal, click, error.

## Visual verification (the agent CAN "see" the app)
`pnpm screenshot` runs `tests/e2e/screenshots.spec.ts`, capturing key states (problem list, solve light/dark, correct verdict, error squiggle, **session setup, learn flashcard**) to `app/screenshots/*.png` (gitignored). **`Read` those PNGs to actually review the rendered UI** ŌĆö this is the agent's stand-in for eyes, and it works in this environment.

**A Playwright Chromium is already available here** (cached at `~/.cache/ms-playwright/chromium-*`) and launches fine ŌĆö verified 2026-07-10 by running the full e2e + screenshot suite green. The old "needs a browser provided by the environment / can't install without root" caveat no longer applies for this setup; just run `pnpm exec playwright test --project=chromium` (or `pnpm screenshot`) with a `pnpm dev` server up on :5173 (the config reuses it). If a *different* environment lacks the browser, the container-baked options below still apply.
- **In the dev container:** bake it in as root at build time ŌĆö `ENV PLAYWRIGHT_BROWSERS_PATH=/ms-playwright` then `RUN npx -y playwright@1.61.0 install --with-deps chromium && chmod -R 777 /ms-playwright`. Or base on `mcr.microsoft.com/playwright:v1.61.0-noble`.
- **On the host (stopgap):** `pnpm exec playwright install chromium && pnpm screenshot`.

**Habit:** after building any UI, run `pnpm screenshot` and Read the PNGs to self-verify layout/new views before reporting done ŌĆö don't default to asking the user to look.

## Commands (run from `app/`)
- `pnpm dev` ŌĆö dev server (http://localhost:5173)
- `pnpm build` ŌĆö typecheck (`tsc --noEmit`) + `vite build`
- `pnpm typecheck` ┬Ę `pnpm lint` ┬Ę `pnpm test` (Vitest)
- `pnpm verify:content` ŌĆö boots the real DuckDB-Wasm engine in Node and grades every question's canonical solution against its dataset (our ADR 0003 authoring check; also our headless stand-in for e2e). Also checks derived metadata isn't stale.
- `pnpm meta:generate` ŌĆö regenerate `src/content/question-metadata.generated.ts` after changing any canonical solution.
- `pnpm coverage` ŌĆö regenerate root `COVERAGE.md`: a live SQL-feature checklist with per-feature question counts, derived from canonical SQL. Feature catalog + detectors in `src/content/features.ts`; `snowflake-vs-duckdb.md` explains the dialect strategy. A question can carry `features: [...]` to tag concepts the detector can't infer (e.g. self-joins).
- `pnpm e2e` ŌĆö Playwright runŌåÆgrade loop in a real browser (`tests/e2e/loop.spec.ts` + `screenshots.spec.ts`). **Works in this environment** (cached Chromium); reuses a running `pnpm dev` on :5173. Run `pnpm exec playwright test --project=chromium`.
- **Definition of done:** typecheck + lint + test + verify:content + build green, **and** the Playwright suite passes with no console errors (ADR 0005). Since the browser works here, run e2e as part of done ŌĆö don't skip it.

## Layout
- `src/grading/` ŌĆö **core IP.** `grade.ts` is the pure output-equivalence comparator (ADR 0004) **and `diffResults`** (structured diff ŌĆö missing/extra rows, column mismatch, order-only failure ŌĆö powering the incorrect-answer view); `grade.test.ts` its unit tests; `types.ts` the `ResultSet`/`Cell` shapes.
- `src/engine/` ŌĆö `duckdb.ts` boots DuckDB-Wasm (lazy, `eh` build) in a worker, loading the wasm+worker from **jsDelivr** (`getJsDelivrBundles`, via a Blob/`importScripts` shim) rather than self-hosting ŌĆö the binaries (~33ŌĆō38 MB) exceed Cloudflare's 25 MiB per-asset cap (ADR 0001 update 2026-07-10); the service worker `CacheFirst`-caches the `@duckdb` CDN path for offline. All connection access is **serialized** through one queue. `runQuery(sql, dialect)` / `validateSql(sql, dialect)` first **transpile** the user's SQL from their dialect to DuckDB via `transpile.ts` (`toDuckDB`, lazy-loaded polyglot wasm ŌĆö ADR 0006; `generic`/`all` pass through). `transpile.ts` also exports `fixupDuckDbSql` ŌĆö a small, pure, unit-tested post-polyglot pass that rewrites the few Snowflake constructs polyglot leaves verbatim but DuckDB spells differently (`TO_VARCHAR`/`TO_CHAR`ŌåÆCAST, `STARTSWITH`ŌåÆ`starts_with`); it's applied in both `toDuckDB` and `verify:content` so authoring and runtime agree. `ensureDataset()` seeds. `result-mapping.ts` is the pure ArrowŌåÆ`ResultSet` mapper reused by the Node verification script.
- `src/content/` ŌĆö typed content model. `types.ts` (Question/Dataset), `datasets/`, `questions/` (one file each), `paths.ts` (curated learning tracks ŌåÆ ordered question ids), `index.ts` registry + `allPacks`/`difficulties`/`allConcepts`/`questionConcepts`/`getMetrics`/`paths`. Concepts (Joins/CTEs/WindowŌĆ”) are derived from each question's metrics (`conceptsOf`). Expected output is **computed at runtime** from each question's canonical solution (ADR 0003) ŌĆö not stored. `recommendNext` (in `index.ts`, unit-tested by `recommend.test.ts`) picks the adaptive "Recommended next" question from progress + concepts.
  - `dialects.ts` ŌĆö a question's `dialects` tag = which SQL dialects it's *appropriate for* (`'generic'` = portable ŌåÆ shown for every dialect; else restricted, e.g. QUALIFY ŌåÆ snowflake/bigquery). `matchesDialect` + `DIALECT_OPTIONS` drive the **dialect picker** on the problem list (persisted in localStorage; filters the list, nav, recommend, and progress totals). This is a **product filter only** ŌĆö execution/grading always uses the single `canonical.generic` reference (true per-dialect syntax fidelity is the still-deferred ADR-0002 transpile layer).
  - `metrics.ts` ŌĆö `extractMetrics` derives structural metadata (tables, joins, window functions, CTEs, aggregates, subqueries, groupBy/orderBy/distinct) from a canonical's parse tree (DuckDB `json_serialize_sql`); pure + unit-tested. `pnpm meta:generate` writes `question-metadata.generated.ts` (committed); `verify:content` fails if it's stale. Don't hand-edit metrics or the generated file ŌĆö regenerate.
- `src/storage/progress.ts` ŌĆö localStorage progress: per-question attempts/solved, "needs review" set (attempted-but-unsolved), and a daily streak. Injectable Storage + clock for tests; migrates the old solved-array format.
- `src/route/useRoute.ts` ŌĆö tiny hash router (no dep): `#/` = list, `#/q/<slug>` = solve, `#/session` = session setup. `navigate()` updates state synchronously (so it's testable) + sets the hash.
- `src/session/session.ts` ŌĆö **practice sessions.** `selectSessionQuestions(candidates, config, {solvedIds, reviewIds}, shuffle?)` builds a disposable, ordered queue of question ids from a user config (pool `all`/`unsolved`/`review` + include-solved top-up ┬Ę sequential(easyŌåÆhard)/random order ┬Ę size). Pure + unit-tested with an injectable shuffle (`session.test.ts`); the app passes `randomShuffle`. App holds the live queue as `SessionState` in localStorage (`ae-practice:session`); progress ("Session ┬Ę n/N") is derived from which queued question is open, so it survives reloads. First cut is minimal ŌĆö no end-of-session summary/timer.
- `src/learn/` ŌĆö **Learn mode (flashcards), the second pillar** alongside Practice (ADR 0007). `leitner.ts` = pure Leitner spaced-repetition scheduler (boxes 1ŌĆō5, intervals ~1┬Ę2┬Ę4┬Ę8┬Ę16 days; "Got it" promotes, "Again" resets to box 1 due today); `store.ts` = localStorage review state (`ae-practice:learn:v1`, injectable storage+clock); `decks/dbt.ts` = ~32 web-verified dbt fundamentals cards; `types.ts`, `index.ts` barrel. All pure logic unit-tested (`leitner.test.ts`, `store.test.ts`). A card review calls `progress.touchStreak()` so both pillars feed the **one shared daily streak**. UI: `components/LearnView.tsx` (flip-card review loop) reached via the top-bar **Practice | Learn** tabs ŌåÆ `#/learn`.
- `src/components/` ŌĆö **two screens**: `ProblemList` (browse table) and `SolveView` (wraps `PracticeView`). Shared `TopBar` (logo + contextual back/prev/next/shuffle nav, circular `ProgressRing` w/ hover tooltip, `ProfileMenu`). `ProfileMenu` houses the theme toggle + a **local-only** sign-in (display name in localStorage ŌĆö no auth backend) so `Avatar` shows real initials. `PracticeView` is the **worksheet**: left = problem + `SchemaPreview` (scrolls); right = full-height `SqlEditor` with a **sliding output drawer** (`Run`/`Submit`/`Reveal` results, collapsible). Plus `SqlEditor`, `SqlBlock`, `ResultsTable`, `DiffView` (renders the structured `diffResults` output on an incorrect submit), `DifficultyBadge`, `formatCell`, `Avatar`, and `SessionSetup` (the "Start a session" config panel ŌĆö builds a queue via `src/session/`). While a session is active the `TopBar` shows a "Session ┬Ę n/N" chip in place of shuffle and prev/next walk the queue.
- `src/editor/` ŌĆö `sqlDialect.ts` (shared DuckDB CodeMirror dialect), `statement.ts` (`statementForCursor` ŌĆö **line-based** selection: the cursor's line picks the query; an empty line picks the previous query ŌĆö built on char-based `statementAt`; plus `statementCount`), `activeStatement.ts` (highlights the selected statement when the worksheet has >1 query). `Ōīś/Ctrl+Enter`, Run, the highlight, and inline validation all use `statementForCursor`.
- `src/theme/useTheme.ts` ŌĆö light/dark theme hook (persisted, defaults to OS); sets `<html data-theme>`.
- `src/App.tsx` ŌĆö routes between `ProblemList`, `SolveView`, `SessionSetup`, and `LearnView` under a shared `TopBar` (with **Practice | Learn** tabs); owns progress/dialect/session/learn state and the session-aware prev/next wiring. `src/App.test.tsx` is a jsdom test of the listŌåÆsolveŌåÆback flow + filters/search/theme + start-sessionŌåÆcounterŌåÆadvance + the Learn flipŌåÆrateŌåÆstreak flow (engine mocked).
- `src/styles.css` ŌĆö **semantic design tokens** (`--bg`/`--surface`/`--accent`/ŌĆ”) defined once for light + `[data-theme='dark']`. Style with tokens, not hardcoded colors, so both themes stay correct (see `notes/research/premium-ux.md`).
- `scripts/verify-content.ts` ŌĆö `pnpm verify:content`; the authoring validation gate (ADR 0003).
- `tests/e2e/` ŌĆö Playwright; drives the real app incl. DuckDB-Wasm in-browser.

## Conventions / gotchas
- The SQL editor is CodeMirror (`SqlEditor`). It renders a contenteditable, so e2e types into `.cm-content` (not a `<textarea>`); the `data-testid="editor"` is on the wrapper.
- DuckDB-Wasm is excluded from Vite dep pre-bundling (`vite.config.ts`); workers/wasm are imported via `?url`.
- Grading is **value-based** ŌĆö column names ignored unless `requireColumnNames`, rows order-insensitive unless `orderMatters`. Don't "fix" a failing question by string-matching SQL.
- To add a question: add a dataset (idempotent `CREATE OR REPLACE` setup SQL) + a `Question` with a runnable canonical solution, and register both in `content/index.ts`.
